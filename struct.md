# 雀魂自动麻将 MOD 架构设计

## 1. 概述

### 1.1 目标
为 STEAM 游戏《雀魂》(Mahjong Soul) 开发一个 MOD，实现自动麻将功能——自动读取牌局状态、AI 决策、自动执行打牌动作。

### 1.2 技术背景
- **游戏引擎**: Unity (IL2CPP 后端)
- **平台**: Windows (STEAM)
- **保护**: 存在反作弊/反调试机制（游戏更新可能引入新保护）
- **推荐 MOD 框架**: BepInEx (IL2CPP 版本) + Harmony

---

## 3. 各层详细设计

### 3.0 Layer 0: MOD 框架层

| 组件 | 说明 |
|------|------|
| **BepInEx (IL2CPP)** | Unity MOD 加载器，IL2CPP 变体，负责注入 DLL、拦截游戏启动流程 |
| **Harmony** | 运行时方法补丁库，hook 游戏内 C# 方法 |
| **Il2CppAssemblyUnhollower** / **Il2CppInterop** | 生成 IL2CPP 托管存根，使 MOD 代码能调用游戏内部类型 |

**职责**:
- 在游戏启动时加载 MOD 程序集
- 提供 Unity 对象遍历与交互能力
- 提供方法 hook 机制

### 3.1 Layer 1A: 数据读取层

负责从运行中的游戏进程获取原始牌局数据。

#### 方案选择（优先级从高到低）

| 方案 | 方式 | 优点 | 缺点 |
|------|------|------|------|
| **A. Unity 对象遍历** | 通过 BepInEx 遍历游戏场景的 GameObject 树，找到牌局相关组件，直接读取字段 | 数据结构完整，准确 | 依赖游戏内部类型，游戏更新可能变动 |
| **B. Memory Scanner** | 扫描进程内存，通过特征码定位关键数据结构 | 不易被反检测 | 实现复杂，维护成本高 |
| **C. 网络包拦截** | hook 网络收发函数，解析协议 | 可获得完整状态 | 协议可能加密，难度大 |
| **D. 视觉识别** | 屏幕截图 + OCR/图像识别 | 通用性强，不易被检测 | 延迟高，准确率不稳定 |

**推荐方案**: **A（Unity 对象遍历）为主 + C（网络包校验）为辅的双源校验**

#### 核心数据读取接口

```csharp
// 牌局状态快照 —— 数据读取层输出的核心数据结构
struct GameStateSnapshot {
    int roundWind;              // 场风 (0=东, 1=南, 2=西, 3=北)
    int selfSeat;               // 自风 (0=东, 1=南, 2=西, 3=北)
    int honba;                  // 本场数
    int[] deposits;             // 供托数 (立直棒)
    int[] scores;               // 四位玩家点数

    Tile[] handTiles;           // 自家手牌 (13/14枚)
    Tile[][] discards;          // 四位玩家的舍牌
    Meld[][] melds;             // 四位玩家的副露
    Tile[] doraIndicators;      // 宝牌指示牌
    Tile uraDoraIndicator;      // 里宝牌指示牌 (立直后可见)

    int selfTurn;               // 当前轮到谁 (0-3)
    int remainingTiles;         // 牌山剩余枚数
    int lastAction;             // 上次动作类型
    Tile lastActionTile;        // 上次动作关联牌
}

// 单张牌
struct Tile {
    int suit;                   // 花色: 0=万, 1=筒, 2=索, 3=字
    int value;                  // 数值: 1-9 (字牌: 1-7对应东南西北白发中)
    bool isRedFive;             // 是否赤5
    bool isDora;                // 是否宝牌
}
```

### 3.2 Layer 1B: 动作执行层

负责将 AI 决策转化为游戏内的实际操作。

#### 方案选择

| 方案 | 实现方式 | 优点 | 缺点 |
|------|----------|------|------|
| **A. Harmony Hook 方法调用** | hook 游戏的动作处理方法，直接调用其内部处理逻辑 | 精确、即时 | 容易被检测 |
| **B. Windows 消息模拟** | SendMessage / PostMessage 模拟鼠标点击按钮区域 | 自然，不易被检测 | 需定位按钮坐标，受UI变动影响 |
| **C. DirectInput / SendInput** | 模拟更底层的输入 | 更真实 | 实现更复杂 |

**推荐方案**: **A（内部方法调用）配合 B（点击模拟降级）**

### 3.3 Layer 2: 牌局状态重建层

将原始数据转换为结构化的麻将逻辑状态，供 AI 决策使用。

```csharp
class GameState {
    // 构造时传入 GameStateSnapshot 进行校验与重建
    static GameState? FromSnapshot(GameStateSnapshot raw);

    // 校验状态一致性（手牌数校验、枚数校验、合法性校验）
    bool Validate();

    // 导出给 AI 引擎的只读视图
    IReadonlyGameState ToImmutable();
}
```

**关键逻辑**:
- **手牌排序与分组**: 按花色分组，按数值排序
- **副露状态**: 记录每个玩家的明牌（碰/吃/杠）
- **牌山剩余计算**: 已知牌驱除后剩余可能牌
- **向听数计算**: 传入当前手牌，输出向听数
- **危险牌追踪**: 记录每张牌被舍出的时机和玩家

#### 3.4.1 打牌效率分析器 (TileEfficiency)

```
输入: 当前手牌 (14枚)
输出: 每张候选切牌的评分

算法流程:
  1. 枚举所有可能的打牌选择 (最多14种)
  2. 对每种选择:
     a. 计算切掉后的向听数 (Shanten)
     b. 计算剩余进张数 (有效牌种类与枚数)
     c. 计算听牌时的良形率 (两面/双碰听牌占比)
     d. 评分 = 进张数 × 权重 + 良形率 × 权重 - 向听数惩罚
  3. 按评分降序排列候选牌
```

- **向听数计算**: 基于"2n+1"公式的标准向听数算法（Mentsu Shanten）
- **牌效率评估**: 参照天凤/麻将AI的标准牌效率理论
- **特殊形判断**: 七对子、国士无双、全不靠等特殊手役判断

#### 3.4.2 防守分析器 (DefenseAnalysis)

```
输入: 对手舍牌信息、自家手牌、局况
输出: 每张牌的"危险度"评分

防守策略等级:
  SAFE (安全牌): 现物（他家切过的牌）、对手立直后自家切过的牌
  SEMI_SAFE (半安全): 筋牌、壁牌、早巡外侧牌
  RISKY (危险): 无筋牌、生牌、中张牌
  DANGEROUS (高危险): 对手自摸切附近的生牌

算法:
  1. 收集所有现物牌 -> 安全度最高
  2. 计算筋牌 (对手舍牌的±3同花色) -> 中等安全
  3. 计算壁牌 (某数值出现4枚的筋) -> 中等安全
  4. 早巡外侧 -> 弱安全
  5. 其余牌按舍出顺位和巡目分级
```

#### 3.4.3 鸣牌决策器 (CallDecision)

```
输入: 对手打出的牌、自家手牌、局况
输出: 是否鸣牌 (吃/碰/杠) 以及鸣牌后的打牌选择

判断因素:
  1. 鸣牌后是否向听前进 (至少减少1向听)
  2. 鸣牌后打点是否充足 (手役价值)
  3. 当前局况是否适合副露 (点数顺位、AL判断)
  4. 鸣牌是否会破坏手牌好形
  5. 是否满贯确等阈值判断
```

#### 3.4.4 立直/和牌决策器

```
ReachDecision:
  - 是否满足立直条件 (门前清、听牌、有1000点棒)
  - 立直判断:
    1. 打点期望值 > 阈值（通常子家5200+，亲家7800+）
    2. 巡目 < T (通常早巡7-8巡内较优)
    3. 听牌良形率 > 30%
    4. 点数顺位压力大时下调阈值
    5. 末期避四需求时不立直

AgariDecision (和牌/跳和不和):
  - 默认可荣和 (最大收益)
  - 特殊场景可跳过:
    1. 亲家连庄利益大于和牌收益（少量/低目）
    2. 故意做更高级手役时（极少见）
```

### 3.5 Layer 4: 策略控制器

高层策略，对战局整体把控。

```
输入: GameState
输出: 策略调整参数 (传递给 Layer 3 各组件)

策略维度:
  - 攻击性 (Aggression): 0.0 (完全弃和) ~ 1.0 (全力进攻)
    影响因素: 手牌强度、巡目、点数顺位、立直状况

  - 速度偏好 (Speed): 0.0 (追求打点) ~ 1.0 (追求快速和牌)
    影响因素: 顺位、AL判断、自家是否领先

  - 风险容忍度 (RiskTolerance): 0.0 (安全至上) ~ 1.0 (高风险高回报)
    影响因素: 局况、点数差、AL状况

子控制器:
  - EarlyGameStrategy:  序盘策略 (1-5巡)
  - MidGameStrategy:    中盘策略 (6-12巡)
  - LateGameStrategy:   终盘策略 (13巡+)
  - AllLastStrategy:    AL(最终局)特殊策略
  - OikakeStrategy:     追立直场景策略
```

## 5. 关键算法与数据结构

### 5.1 向听数计算 (Shanten)

```csharp
// 核心算法: 基于拆解的向听数
int CalculateShanten(Tile[] hand) {
    // 标准手: 2n+1 公式
    // 七对子: 7 - pairs + max(0, 6 - pairs)
    // 国士无双: 13 - uniqueTerminalsHonors - hasPair
    // 返回三者的最小值
}
```

**实现方案**:
1. 递归拆解算法 (标准手): 枚举所有面子拆解方式，取最大搭子数
2. 查表法 (优化): 手牌编码为 34 维向量，预计算向听数表（约 34^5 状态，可通过 DP 压缩）
3. **推荐**: 查表法 + 递归剪枝相结合，平衡速度和灵活性

### 5.2 手牌编码

```csharp
// 34种牌的一维编码

// 18-26: 一索~九索
// 27-33: 东南西北白发中
// 每牌最多4枚
```

## 8. 风险与注意事项

### 技术风险
| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| 游戏版本更新导致数据偏移 | 高 | 高 | 采用配置化偏移量，支持快速适配 |
| IL2CPP hook 不稳定 | 中 | 高 | 准备输入模拟作为备选执行方案 |
| 反作弊系统升级 | 中 | 中 | 保守操作，不做侵入性 Hook |
| 手牌数据加密/混淆 | 中 | 高 | 网络包校验 + Unity 多层次读取 |

### 法律与合规风险
1. **游戏条款**: 雀魂用户协议通常禁止第三方辅助程序
2. **STEAM VAC**: STEAM 反作弊系统可能导致封号
3. **建议**: 此项目仅供学习研究使用，请勿在正式账号上使用
